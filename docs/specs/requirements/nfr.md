# 非功能需求规格 (NFR)

> 版本：v1.0  
> 状态：Draft

---

## 1. 性能需求

### NFR-PERF-001: 网关延迟

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| P50 延迟 | < 20ms | 不含上游响应时间，仅网关处理 |
| P95 延迟 | < 50ms | |
| P99 延迟 | < 200ms | |

**测试条件:**
- 单节点部署
- 100 RPS 持续 5 分钟
- 非流式请求

### NFR-PERF-002: 吞吐量

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 单节点 RPS | ≥ 1000 | 100 并发，持续 1 分钟 |
| 连接数 | ≥ 10000 | 长连接场景 |

### NFR-PERF-003: 资源占用

| 指标 | 目标值 | 场景 |
|------|--------|------|
| 内存 | < 512MB | 空闲状态 |
| 内存 | < 1GB | 1000 并发连接 |
| CPU | < 50% | 500 RPS |

## 2. 可用性需求

### NFR-AVAIL-001: 服务可用性

| 指标 | 目标值 | 计算方式 |
|------|--------|----------|
| 月度可用性 | ≥ 99.5% | (总时间 - 故障时间) / 总时间 |
| 最大故障时间 | < 3.65 小时/月 | |

### NFR-AVAIL-002: 故障恢复

| 场景 | 恢复时间目标 (RTO) |
|------|-------------------|
| 单节点重启 | < 30s |
| 数据库故障 | < 5min (降级运行) |
| Redis 故障 | < 1min (降级到内存模式) |

### NFR-AVAIL-003: 降级策略

| 组件故障 | 降级行为 |
|----------|----------|
| Redis 不可用 | 降级到内存模式，仅计费延迟持久化 |
| DB 不可用 | 流水暂存内存/文件，DB 恢复后补写 |
| 单供应商故障 | 该供应商模型返回 503，其他正常 |

## 3. 安全需求

### NFR-SEC-001: 认证安全

| 要求 | 实现 |
|------|------|
| API Key 存储 | SHA256 哈希，不存明文 |
| 传输加密 | 强制 HTTPS (TLS 1.2+) |
| Key 格式 | 42 字符，含前缀便于识别 |

### NFR-SEC-002: 访问控制

| 要求 | 实现 |
|------|------|
| 用户隔离 | 每个用户独立余额和 Key |
| 管理接口 | 独立 Admin Key 认证 |
| 速率限制 | 默认 60 RPM / Key |

### NFR-SEC-003: 数据安全

| 要求 | 实现 |
|------|------|
| 对话内容 | **不存储**，仅透传 |
| 上游 Key | 环境变量注入，不入库 |
| 敏感日志 | 脱敏处理 (Key 仅显示前缀) |

### NFR-SEC-004: 防护措施

| 威胁 | 防护 |
|------|------|
| DDoS | Cloudflare WAF + 速率限制 |
| 暴力破解 | Key 格式校验 + IP 黑名单 |
| 注入攻击 | 参数化查询 + 输入校验 |

## 4. 可扩展性需求

### NFR-SCALE-001: 水平扩展

| 要求 | 说明 |
|------|------|
| 无状态设计 | 应用层无本地状态，支持多实例 |
| 共享存储 | Redis + DB 集中存储 |
| 负载均衡 | 支持 L7 负载均衡器 |

### NFR-SCALE-002: 数据容量

| 数据类型 | 初期容量 | 扩展方案 |
|----------|----------|----------|
| 用户数 | 1000 | SQLite → PostgreSQL |
| 流水数 | 100万/月 | 按月分表 + 定期归档 |
| Redis 数据 | 1GB | Redis Cluster |

## 5. 可维护性需求

### NFR-MAINT-001: 日志规范

```json
{
  "timestamp": "2026-04-22T10:00:00Z",
  "level": "INFO",
  "request_id": "req_abc123",
  "user_id": "user_001",
  "method": "POST",
  "path": "/v1/chat/completions",
  "model": "claude-3-5-sonnet-20241022",
  "duration_ms": 1234,
  "status": 200,
  "input_tokens": 100,
  "output_tokens": 200,
  "cost": 0.003960
}
```

### NFR-MAINT-002: 监控指标

| 指标类型 | 指标名 | 说明 |
|----------|--------|------|
| Counter | `http_requests_total` | 请求总数 |
| Histogram | `http_request_duration_seconds` | 请求延迟分布 |
| Gauge | `active_connections` | 当前连接数 |
| Counter | `upstream_errors_total` | 上游错误数 |
| Gauge | `user_balance` | 用户余额 |
| Counter | `tokens_consumed_total` | Token 消耗总数 |

### NFR-MAINT-003: 告警规则

| 条件 | 级别 | 动作 |
|------|------|------|
| 错误率 > 5% (5min) | Warning | Slack 通知 |
| 错误率 > 20% (5min) | Critical | 电话告警 |
| P99 > 5s (5min) | Warning | Slack 通知 |
| Redis 断连 | Critical | 电话告警 |
| 用户余额 < $1 | Info | 邮件提醒用户 |

## 7. 可观测性需求

### NFR-OBS-001: 链路追踪

| 要求 | 实现 |
|------|------|
| 请求 ID | 每个请求生成唯一 ID，贯穿全链路 |
| 上游关联 | 记录上游请求 ID |
| 日志关联 | 所有日志包含 request_id |

### NFR-OBS-002: 健康检查

**端点:** `GET /health`

**响应:**
```json
{
  "status": "healthy",
  "version": "1.0.0",
  "uptime": "72h30m15s",
  "components": {
    "redis": {"status": "healthy", "latency_ms": 1},
    "database": {"status": "healthy", "latency_ms": 5}
  }
}
```

**状态定义:**
| status | 条件 |
|--------|------|
| healthy | 所有组件正常 |
| degraded | 有组件异常但服务可用 |
| unhealthy | 服务不可用 |
