# Sub2API 项目概述

> 版本：v1.0  
> 最后更新：2026-04-22  
> 状态：Draft

---

## 1. 项目定位

**Sub2API** 是一个轻量级 API 聚合中转平台，提供 OpenAI 兼容格式的统一接口，支持多供应商 API 接入、Token 计费与用户配额管理。

## 2. 核心价值主张

| 用户痛点 | 解决方案 |
|----------|----------|
| 海外 API 国内难以直接访问 | 提供稳定的中转服务 |
| 单个供应商有速率限制 | 多 Key 池化，负载均衡 |
| 需要分别注册多个平台 | 一站式接入，统一格式 |
| Token 计费不透明 | 实时用量追踪，透明计费 |

## 3. 目标用户

| 用户类型 | 特征 | 核心需求 |
|----------|------|----------|
| **个人开发者** | 使用 NextChat/LobeChat 等客户端 | 稳定访问、按量付费 |
| **小型团队** | 多人共用额度 | 统一账户、分 Key 管理 |
| **应用开发者** | 集成 AI 能力到产品 | API 稳定性、计费透明 |

## 4. 技术选型

| 组件 | 选型 | 决策理由 |
|------|------|----------|
| **后端语言** | Go (Gin) | 高并发、低内存、单二进制部署 |
| **缓存层** | Redis | 原子性扣费、高性能计数 |
| **数据库** | SQLite → PostgreSQL | 初期简单，后期可扩展 |
| **前端框架** | Next.js | SSR 支持、Vercel 免费部署 |
| **支付** | Stripe Checkout | 托管支付页面，降低合规风险 |
| **反向代理** | Caddy | 自动 HTTPS 证书 |
| **容器化** | Docker Compose | 一键部署 |

## 6. 质量属性要求

| 属性 | 目标 | 度量方式 |
|------|------|----------|
| **可用性** | 99.5% uptime | 月度可用时间 / 总时间 |
| **延迟** | P99 < 200ms (不含上游) | 网关处理延迟 |
| **并发** | 1000 RPS / 单节点 | 压测峰值 |
| **数据一致性** | 余额不超扣 | 并发测试无负余额 |
| **安全性** | API Key 哈希存储 | 数据库无明文 Key |

## 7. 约束条件

### 7.1 技术约束

- 必须兼容 OpenAI API 格式（`/v1/chat/completions`）
- 支持 SSE 流式响应透传
- 单节点内存占用 < 512MB

### 7.2 业务约束

- 遵守上游供应商 ToS（不存储对话内容）
- 支付仅支持 Stripe（初期）

### 7.3 时间约束

| 阶段 | 时间 | 交付目标 |
|------|------|----------|
| Phase 1: MVP | 1 周 | 单供应商可商用 |
| Phase 2: 计费 | 1 周 | Redis 原子扣费 + DB 持久化 |
| Phase 3: 后台 | 2 周 | 用户自助 Dashboard |
| Phase 4: 支付 | 2 周 | Stripe 集成 |
| Phase 5: 规模化 | 持续 | 多节点 + 监控 |
